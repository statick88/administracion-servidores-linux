Unidad 11: MariaDB en Producción

Introducción

MariaDB es el reemplazo directo de MySQL en la mayoría de distribuciones Linux. Es una base de datos relacional utilizada por millones de aplicaciones web. En esta unidad vas a dominar la instalación, configuración de producción, usuarios y privilegios, backups, y acceso remoto seguro.

Configuración de Producción

Archivo de Configuración

BASH
# Ver configuración actual
$ sudo mariadb -e "SHOW VARIABLES LIKE 'max_connections';"
+--------------------+
| max_connections    |
+--------------------+
| 151                |
+--------------------+

# Archivo de configuración
$ ls -la /etc/mysql/
mariadb.conf.d  my.cnf
1
Config files: /etc/mysql/mariadb.conf.d/50-server.cnf

Configuración de Producción

BASH
$ sudo tee /etc/mysql/mariadb.conf.d/99-production.cnf <<'EOF'
[mysqld]
# Conexiones
max_connections = 200
wait_timeout = 60
interactive_timeout = 60

# InnoDB
innodb_buffer_pool_size = 1G          # 70% de RAM disponible
innodb_log_file_size = 256M
innodb_flush_log_at_trx_commit = 2     # Sécurité + rendimiento
innodb_flush_method = O_DIRECT

# Logs
log_error = /var/log/mysql/error.log
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2

# Character set
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci

# Seguridad
local_infile = 0
skip_symbolic_links = 1
bind-address = 127.0.0.1               # Solo localhost por defecto

[client]
default-character-set = utf8mb4
EOF

$ sudo systemctl restart mariadb
1
Production config: Optimizado para producción

Usuario con Múltiples Hosts

BASH
# Usuario para desarrollo (localhost)
CREATE USER 'devuser'@'localhost' IDENTIFIED BY 'DevPass123!';
GRANT ALL PRIVILEGES ON appdb.* TO 'devuser'@'localhost';

# Usuario para aplicación (servidor web)
CREATE USER 'appuser'@'10.0.1.0/255.255.255.0' IDENTIFIED BY 'AppNetworkPass!';
GRANT SELECT, INSERT, UPDATE, DELETE ON appdb.* TO 'appuser'@'10.0.1.0/255.255.255.0';

1
Múltiples hosts: Por IP o red específica

BASH
# Backup de base de datos específica
$ sudo mysqldump -u root -p appdb > /backup/appdb-$(date +%Y%m%d).sql

# Backup de todas las bases de datos
$ sudo mysqldump -u root -p --all-databases > /backup/all-databases-$(date +%Y%m%d).sql

# Backup con gzip (comprimido)
$ sudo mysqldump -u root -p appdb | gzip > /backup/appdb-$(date +%Y%m%d).sql.gz

Base específica: Un archivo por DB
2
Todas: Un archivo con todas las DB

MariaDB Backup (Físico)

BASH
# Instalar MariaDB Backup tool
$ sudo apt install -y mariadb-backup

# Backup físico
$ sudo mariabackup --backup \
    --target-dir=/backup/full-$(date +%Y%m%d) \
    --user=root --password='StrongRootPass-2026!'
1

# Restaurar物理 backup
$ sudo mariabackup --prepare \
    --target-dir=/backup/full-20260331

$ sudo systemctl stop mariadb

$ sudo mariabackup --restore \
    --target-dir=/backup/full-20260331 \
    --user=root --password='StrongRootPass-2026!'

$ sudo chown -R mysql:mysql /var/lib/mysql
$ sudo systemctl start mariadb
1
mariabackup: Backup rápido para grandes DB
2
Restaurar: Pasos para restaurar

Cronjob para Backups Automáticos

BASH
# Crear script de backup
$ sudo tee /usr/local/bin/backup-mariadb.sh <<'EOF'
#!/bin/bash
DATE=$(date +%Y%m%d)
BACKUP_DIR="/backup"
DB_NAME="appdb"
DB_USER="backupuser"
DB_PASS="BackupUserPass123!"

# Crear directorio si no existe
mkdir -p $BACKUP_DIR

# Realizar backup
mysqldump -u $DB_USER -p"$DB_PASS" $DB_NAME | gzip > $BACKUP_DIR/${DB_NAME}-${DATE}.sql.gz

# Eliminar backups de más de 7 días
find $BACKUP_DIR -name "*.sql.gz" -mtime +7 -delete

# Notificar (opcional)
echo "Backup completado: $DATE" | logger -t backup-mariadb
EOF

$ sudo chmod +x /usr/local/bin/backup-mariadb.sh

# Agregar a crontab
$ sudo crontab -e
# Agregar línea:
# 0 3 * * * /usr/local/bin/backup-mariadb.sh
1
Cronjob diario: Backup automático a las 3 AM

Casos de Estudio

Caso 1: Base de Datos Lenta

Síntomas:

BASH
$ mariadb -u appuser -p -e "SELECT * FROM orders WHERE date > '2026-01-01'"
# Tarda 30+ segundos

Diagnóstico:

BASH
# Ver queries lentos
$ sudo tail -50 /var/log/mysql/slow.log
2

# Ver EXPLAIN
EXPLAIN SELECT * FROM orders WHERE date > '2026-01-01';
2
Slow query log: Ver queries problemáticos
3
EXPLAIN: Analizar plan de ejecución

Solución:

BASH
# Agregar índice
ALTER TABLE orders ADD INDEX idx_date (date);

# Verificar
EXPLAIN SELECT * FROM orders WHERE date > '2026-01-01';

Caso 2: Too Many Connections

Problema:

BASH
$ mariadb -u appuser -p
ERROR 1040 (08004): Too many connections

Diagnóstico:

BASH
# Ver conexiones actuales
$ mariadb -e "SHOW STATUS LIKE 'Threads_connected';"
+---------------------+
| Threads_connected   |
+---------------------+
| 151                 |
+---------------------+

# Ver max
$ mariadb -e "SHOW VARIABLES LIKE 'max_connections';"
+--------------------+
| max_connections    |
+--------------------+
| 151                |
+--------------------+
2
max_connections: Solo 151

Solución:

BASH
# Aumentar conexiones (agregar a my.cnf)
# [mysqld]
max_connections = 300

$ sudo systemctl restart mariadb

Caso 3: Contraseña Root Olvidada

Problema: No se puede conectar como root.

Solución:

BASH
# Detener MariaDB
$ sudo systemctl stop mariadb

# Iniciar en modo seguro
$ sudo mysqld_safe --skip-grant-tables --skip-networking &

# Conectar sin password
$ sudo mariadb

# Resetear password
FLUSH PRIVILEGES;
ALTER USER 'root'@'localhost' IDENTIFIED BY 'NewRootPass2026!';

# Detener y reiniciar normal
$ sudo killall mysqld
$ sudo systemctl start mariadb
1
Modo safe: Sin privilegios, permite reset

Caso 4: Recovery Después de Crash

Problema: Base de datos no inicia después de apagón.

Diagnóstico:

BASH
$ sudo systemctl start mariadb
Job for mariadb.service failed because a timeout was exceeded.

# Ver logs
$ sudo journalctl -u mariadb -n 50

Solución:

BASH
# Recovery mode
$ sudo mysqld --innodb-force-recovery=1 &

# Verificar tablas
$ sudo mariadb -e "CHECK TABLE dbname.tablename;"

# dump y restore si es necesario
$ sudo mysqldump -A > /backup/before-fix.sql

# Detener y reiniciar normal
$ sudo killall mysqld
$ sudo mysqld_safe --innodb-recovery &

# Luego restaurar
$ sudo mariadb < /backup/before-fix.sql

Health Check

BASH
# Script de health check
#!/bin/bash
RESULT=$(mariadb -u root -p'STRONG_PASSWORD' -e "SELECT 1" 2>/dev/null)
if [ "$RESULT" = "1" ]; then
    echo "MariaDB OK"
    exit 0
else
    echo "MariaDB FAIL"
    exit 1
fi
